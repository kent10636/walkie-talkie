# 下次恢复工作 Checklist

**使用场景**: 当你手边有测试设备（模拟器或真机）时，按照本清单快速恢复并验证。

## 准备环境

- [ ] 更新代码到最新（如果有远程仓库）
- [ ] 检查 `docker` 是否可用

## 启动顺序

### 1. 基础设施
```bash
docker compose up -d
```
验证:
- LiveKit 在 `ws://localhost:7880` 可访问
- 可选：访问 http://localhost:7880 查看 LiveKit 管理界面（如果开启）

### 2. 后端
```bash
cd server
go mod tidy
LIVEKIT_API_KEY=devkey LIVEKIT_API_SECRET=secret go run cmd/server/main.go
```
验证:
- 后端在 `:8080` 启动
- `/health` 接口返回 `{"status":"ok"}`

### 3. Android 客户端
1. Android Studio 打开项目根目录下的 `android/` 文件夹
2. 等待 Gradle Sync 完成
3. 选择模拟器或真机运行

**重要配置提醒**:
- 模拟器默认使用 `10.0.2.2:8080`（已在代码中）
- 真机测试时，修改 `android/app/src/main/java/com/walkietalkie/app/data/ApiClient.kt` 中的 `BASE_URL`

## 核心验证流程（最小 Demo）

推荐使用**两台设备**同时操作：

### 步骤 1: 昵称
- 两台设备分别输入不同昵称（如 "Kent" 和 "测试机"）

### 步骤 2: 创建/加入群组
**方式 A（推荐）**:
- 设备A 创建群组（记录生成的 6 位码）
- 设备B 使用相同码加入

**方式 B**:
- 两台设备都创建自己的群（不同码，不推荐用于首次测试）

### 步骤 3: 进入房间
- 两台设备都进入同一个房间
- 观察状态文字是否变为 "已连接"

### 步骤 4: PTT 测试（最关键）
1. 设备A **长按** 大圆形按钮
2. 设备A 说话（正常音量）
3. 设备B 是否能听到声音？
4. 松开按钮
5. 声音是否停止？
6. 反向测试（设备B 说话，设备A 听）

### 步骤 5: 其他基础检查
- [ ] 成员列表是否显示两个昵称
- [ ] 点击“离开群组”是否正常返回
- [ ] 重新加入是否还能正常对讲

## 常见问题排查

| 问题 | 可能原因 | 解决建议 |
|------|----------|----------|
| 无法连接后端 | IP 地址不对 | 检查 ApiClient.kt 中的 BASE_URL |
| 连接 LiveKit 失败 | LiveKit 容器未启动或地址错误 | 确认 `docker compose up -d` 已执行 |
| 按住按钮没声音 | 麦克风权限未授予 | 在设备设置中手动授予 App 麦克风权限 |
| 只有一方能听到 | 网络或 LiveKit 配置问题 | 检查两台设备是否在同一网络，查看后端日志 |
| 代码编译失败 | Gradle 依赖问题 | Invalidate Caches / Restart + Sync |

## 下次迭代建议优先级

1. **必须先解决**:
   - 运行时请求麦克风权限
   - 连接失败时的友好提示
   - 确认基本语音双向通话

2. **紧接着**:
   - 房间内实时参与者列表更新
   - 说话状态指示（谁在说话）
   - 离开时正确断开 LiveKit 连接

3. **后续增强**:
   - 后端持久化 + 群组历史
   - 蓝牙耳机支持
   - UI 优化（按下反馈、波形动画）
